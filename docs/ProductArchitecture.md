# **Product & Architecture Specification Document: Family Kindness Tracker**

This document establishes the production-ready functional and technical specification for the Minimum Viable Product (MVP) of the Family Kindness Tracker.

## **1. Executive Summary & Core Value Proposition**

The primary objective of this application is to cultivate and increase intentional acts of kindness within a household. The system acts as a shared, positive-reinforcement tool where family members collectively contribute toward a unified goal.

### **User Persona Matrix**

  * **Authenticated Admins (Parents):** Hold elevated permissions. They interact with the system to log their own actions, log witnessed acts, view the core dashboard, and audit, edit, or delete historical entry data.
  * **Unauthenticated Users (Children):** Interact with the system via an open, low-friction shared interface. They log witnessed acts of kindness performed by other family members and view the core progress dashboard.

## **2. Core Application States & Routing Boundaries**

The system operates across two separate routing layers and security zones:

### **A. Unauthenticated Public Router (`/dashboard`)**

  * Accessible by default on any household device without inputting credentials.
  * Provides read-only access to the public family kindness meter and the total completions counter.
  * Provides write-only access to the core log submission form (protected by UI guardrails).

### **B. Authenticated Administrative Router (`/admin`)**

  * Strictly protected behind an authentication barrier requiring parent login credentials.
  * Inherits all public dashboard capabilities.
  * Grants full read, update, and delete (CRUD) privileges over all raw history logs generated by the system.

## **3. User Workflows**

### **Workflow A: Child Submission (Unauthenticated)**

1.  **Entry:** The child approaches the shared application interface (such as a tablet or household computer running the app).
2.  **Identity Selection:** The child selects their own identity from a "Who is submitting" dropdown list.
3.  **Guardrail Execution:** The system dynamically filters the choices for the "Who is this for" dropdown list, entirely removing the child's selected name from the options.
4.  **Completion:** The child selects the corresponding kindness category, types an optional description, and clicks submit.
5.  **Success:** The backend validates the payload, updates the collective meter, and returns the user to the blank dashboard screen.

### **Workflow B: Parent Management & Oversight (Authenticated)**

1.  **Authentication:** The parent accesses the secure /admin route and logs in.
2.  **Audit:** The parent views a chronological data table containing every raw kindness submission entry.
3.  **Correction:** If a child enters an inappropriate description or submits duplicate data, the parent selects the entry to either modify the text fields or permanently delete the record.
4.  **Sync:** The database reflects the change instantly, and the server re-aggregates the collective family meter values.

## **4. Data Ingestion, Types & Strict Constraints**

The system relies on a strictly typed data ingest schema to maintain referential integrity and lean database payloads.

### **Entity Definitions**

#### **1. User / Family Member**

  * **ID:** Unique Identifier (Primary Key / Object Identifier).
  * **Name:** String. Displayed across selection dropdowns.
  * **Role:** Enum (Admin or Standard).

#### **2. Kindness Log Entry**

  * **ID:** Unique Identifier (Primary Key / Object Identifier).
  * **SubmittedBy:** User ID reference (Foreign Key).
  * **Beneficiary:** User ID reference (Foreign Key).
  * **Category:** Strict Enum matching the Points Matrix.
  * **PointsAwarded:** Integer. Calculated immediately upon submission.
  * **Description:** String. Strict limit of **200 characters maximum**.
  * **Timestamp:** DateTime. Auto-generated on successful server insertion.

### **Ingestion Validation Rules**

  * **Predefined Selectors:** "SubmittedBy", "Beneficiary", and "Category" fields must map strictly to pre-populated system values. Raw text inputs are completely blocked for these properties.
  * **Data Restrictions:** The application blocks any payload where SubmittedBy is identical to the Beneficiary. This rule is enforced universally for every submission, regardless of the submitting user's role (the validation schema has no awareness of role).

## **5. Server-Side Business Logic & Calculations**

All point calculations, aggregations, and data modifications take place securely on the backend API layer to guarantee data consistency.

### **Points Matrix Definition**

When a payload is processed, the server evaluates the chosen category string and binds an immutable point score to the entry based on the following deterministic rules:

| Category Enum String | Point Value          | Rules / Variations                            |
| :------------------- | :------------------- | :-------------------------------------------- |
| Kind Words           | 10 pts               | Fixed baseline                                |
| Showing Gratitude    | 15 pts               | Fixed baseline                                |
| Helping Hand         | 20 pts               | Fixed baseline                                |
| Other                | 5, 10, 15, or 20 pts | Must include client-selected integer override |

### **The Kindness Meter Calculation Logic**

The application tracks progress using a **Collective Family Meter** based on a fixed maximum threshold (for example, 1,000 points representing a "Full Meter").

  * **Total Points Accumulation:** Every time an act of kindness is logged, the server grabs the point value for that category and adds it to a grand running total of all points ever earned by the family.
  * **Total Completions Counter:** To calculate how many times the meter has been filled, the server takes the grand running total and divides it by the 1,000-point threshold, rounding the result down to the nearest whole number.
  * **Current Meter Fill Percentage:** To calculate how full the current progress bar is, the server looks only at the remainder points earned since the last 1,000-point milestone was crossed. It takes those remainder points, divides them by 1,000, and multiplies by 100 to get a clean percentage from 0% to 99%.

## **6. Data Retrieval & User Interface Requirements**

The user interface emphasizes radical simplicity and visual reinforcement.

### **Layout Requirements (Main Dashboard View)**

  * **Primary Focal Point:** A highly visible graphical "Kindness Meter" displaying the calculated fill percentage.
  * **Milestone Indicator:** If the Total Completions Counter is greater than zero, a bold text readout must display prominently: *"We have filled our Kindness Meter \[X\] times\!"*
  * **Primary Action Hook:** A clean, accessible button to launch the Log Submission form.

### **Data Security & Read Filters**

  * **Public Reads:** The dashboard reads only the aggregated meter statistics (Percentage and Completion Count). Individual user scoreboards or historical descriptions are not exposed on the unauthenticated dashboard to maintain privacy.
  * **Admin Reads:** The administrative panel reads an unbroken, chronological table or list format of all historical logs for audit purposes.

## **7. Access Control & System Architecture Boundaries**

### **Authentication Layout**

  * **Public Domain:** HTTP GET /api/meter-status and HTTP POST /api/logs (specifically matching standard/child workflows) require no active session tokens.
  * **Protected Domain:** All administrative pathways (HTTP PUT /api/logs/:id, HTTP DELETE /api/logs/:id, and HTTP GET /api/admin/logs) require a validated administrative session token.

### **Explicitly Out of Scope for MVP**

To combat scope creep and preserve an ultra-lean deployment timeline, the following elements are explicitly barred from development in Version 1.0:

1.  **Individual Performance Tracking:** No individual user scoreboards or personal historical trends.
2.  **Milestone-Driven Automation Triggers:** No actions, automated emails, external service calls, or code executions when the counter increments or when the meter hits 10 completions.
3.  **Rich Media Attachments:** No photo uploads, video attachments, or voice memos within the kindness log entries.
4.  **External Notification Loops:** No push notifications, SMS alerts, or emails dispatched upon new submissions.
